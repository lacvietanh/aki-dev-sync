### P0 — nguyên nhân trực tiếp của "Apply không ăn"

**P0-1. `selectedTargets` mặc định chỉ `['ag']`, không persist**
`ClaudeSettingModal.vue:419`: `const selectedTargets = ref(['ag'])`. Checkbox "claude" không tick sẵn, và không lưu localStorage nên mỗi lần mở modal lại reset. - **Fix:** mặc định `['ag', 'cc']`; persist vào localStorage (key riêng, ví dụ `aki-statusline-targets`); disable nút Apply khi `selectedTargets.length === 0`; tooltip nút Apply phản ánh đúng target đang chọn.

**P0-2. Field `account` trên CC luôn rỗng — thiếu disk fallback**
Generator CC chỉ đọc `.account.email // .user.email // .email` từ stdin, mà payload Claude Code **không có** field email (research doc §2.2 đã ghi rõ). Bước 2 của "Dual-Strategy" (fallback `~/.claude.json` → `.oauthAccount.emailAddress`) **chưa được viết vào generator**. Toggle Account trên UI là nút chết đối với CC.
- **Fix:** trong `generate_cc_reference_script`, khi `account` enabled, sinh thêm shell:
  ```sh
  if [ -z "$account_email" ] && [ -f "$HOME/.claude.json" ]; then
    account_email=$(jq -r '.oauthAccount.emailAddress // ""' "$HOME/.claude.json" 2>/dev/null)
  fi
  ```

**P0-3. Field `account` trên AGY — fallback jq sai shape**
`statusline.rs:849`: `jq -r '.[0].email // ""' ~/.gemini/google_accounts.json` — file thật là object `{"active": ..., "old": [...]}`. Đã chạy thử: `jq: error Cannot index object with number` (bị `2>/dev/null` nuốt) → account không bao giờ hiện qua fallback.
- **Fix:** `jq -r '.active // ""'`. Lưu ý: fallback disk chỉ phản ánh account "active toàn cục", đúng tinh thần degrade-gracefully của research doc; nguồn chính vẫn là `.account.email // .email` từ payload RAM của process.

**P0-5. rlcache hiển thị quota MA — lỗ hổng thiết kế đã release từ 1.10.0 đến 1.17.0 (data integrity)**

> Ưu tiên cao hơn mọi bug hiển thị còn lại trong plan này: đây không phải "hiển thị xấu" mà là **hiển thị sai số liệu quota** — người dùng đọc một con số không thuộc về mình và tin nó.

**Bằng chứng thực địa.** Trên remote host `bien`, statusline hiện `7d 45%` trong khi tài khoản đang dùng **không hề có weekly limit** — `/usage` của chính tài khoản đó không có mục 7 ngày.

**Không phải regression mới.** Xác nhận bằng `git log -p --follow -- share/aki-statusLine/statusline.sh`:
- Cơ chế "aki-rlcache v2" (cache toàn bộ `rate_limits` xuống `~/.claude/rate-limits-cache.json`, merge lại khi payload thiếu field) vào từ **1.10.0** (commit `e461460`, 2026-07-15) và **không đổi logic** kể từ đó — commit gần nhất chạm file (`4836696`, lúc `package.json = 1.15.1`) chỉ đổi ký tự em-dash.
- Tag release gần nhất là **1.17.0**, vẫn mang nguyên rlcache v2 của 1.10.0.
- ⇒ Lỗ hổng đã **release và tồn tại xuyên suốt 1.10.0 → 1.17.0** (ít nhất), **không** phải hệ quả của bản v3 đang sửa dở cho 1.18.0 (chưa release).

**Root cause** (đúng cho cả v2 *và* bản v3 hiện tại trong `share/aki-statusLine/statusline.sh` với merge `$old * $new`):
1. **Không có TTL / không kiểm tra hạn** — không hề đối chiếu `resets_at`; entry đã qua thời điểm reset từ lâu vẫn được hiển thị. `rate_block` khi hết hạn chỉ ẩn phần **ETA**, **không ẩn %**.
2. **Không khoá theo account** — file cache dùng chung cho *mọi* account từng chạy trên cùng host. Object-merge chỉ ghi đè/thêm key **có mặt** trong live payload, **không bao giờ xoá** key vắng mặt ⇒ field nào lọt vào cache một lần là sống gần như vĩnh viễn.

⇒ `45% 7d` trên host `bien` nhiều khả năng là **rác từ một account/session KHÁC** từng chạy trên cùng host trong khoảng 1.10.0–1.17.0, bị cache giữ lại tới tận bây giờ.

**Đề xuất fix:**
- Không merge/giữ bất kỳ field nào có `resets_at` **đã ở quá khứ** — loại khỏi cache ngay mỗi lần ghi lại (coi là hết hạn).
- Khoá cache theo account: hoặc key theo `account_email` ngay trong file cache, hoặc **xoá toàn bộ cache** khi phát hiện account hiện tại (`~/.claude.json` → `.oauthAccount.emailAddress`) khác account của lần cache trước.

**Trạng thái: ĐÃ FIX (rlcache v4, 22/07)** — trong `generate_cc_reference_script`:
- Gate account: đọc `~/.claude.json` → `.oauthAccount.emailAddress`, so với field `.account` trong file cache; **không khớp → không đọc cache** (cache của account khác, hoặc cache legacy 1.10.0–1.17.0 không có field `account` trong khi máy đang có account → coi như không tin được). File cache mới luôn ghi kèm `"account"`.
- Prune theo hạn: mọi entry có `resets_at` đã ở quá khứ bị loại **trước khi** hiển thị và trước khi ghi lại. `resets_at == 0` = "không biết", **không** phải "hết hạn" → giữ nguyên (đúng hành vi test cũ).
- Nếu sau 2 gate không còn gì → `rm -f` file cache, không để lại file cũ cho lần gọi sau đọc lại và tin lần nữa.
- Test mới: `cc_drops_a_cached_quota_whose_reset_has_passed`, `cc_ignores_a_cache_written_by_another_account`.
- Verify thực tế (chạy script sinh ra dưới `$HOME` sandbox, 7 case): 7d hết hạn bị loại khỏi cả dòng lẫn file cache; cache của account khác bị bỏ qua; cache legacy không có `account` bị bỏ qua khi máy có account; account khớp + chưa hết hạn vẫn restore đủ 5h/7d; `resets_at:0` vẫn giữ; file ghi ra luôn là JSON hợp lệ kể cả khi không có account (`"account":""`) và round-trip được ở lần gọi kế; cache hỏng JSON không làm trắng dòng.

**P1-2. File share + file đã cài mất hẳn `7d`**
Bản sửa tay có `g_group_quota=$(join_with " " "$g_rate_limits_5h")` — tính `g_rate_limits_7d` xong **không đưa vào group** → statusline đang chạy trên máy không bao giờ hiện quota 7 ngày. Ngoài ra vòng join tham chiếu `$g_rate_reset_5h` không tồn tại, và không có field account (19 field, thiếu `account_email`).
- **Fix (đã làm, 1.18.0):** **xoá hẳn `share/aki-statusLine/`** thay vì tìm cách giữ nó đồng bộ. Đó là bản sao chép tay của thứ generator vốn đã sinh ra, không code nào load lúc chạy — chính nó là cái bẫy khiến một đợt fix đi vào file chết. README bỏ luôn section giới thiệu. Từ đây generator là nguồn duy nhất, chỉ còn 2 file: F1 (generator) → F3 (`~/.claude/statusline-command.sh` trên từng máy).

**P1-3. AGY: quota dạng số bị tính sai công thức → `-2400%`**
`(1 - rem) * 100` chỉ đúng khi `rem` là fraction 0..1. Payload dạng `{"gemini-5h": 25}` (nhánh `type == "number"`) → `(1-25)*100 = -2400%`. Unit test `agy_renders_a_line` in ra đúng chuỗi lỗi này nhưng vẫn pass vì chỉ assert email.
- **Fix:** trong shell sinh ra: nếu `rem <= 1` → fraction; nếu `> 1` → coi là **percent remaining** → `used = 100 - rem`. Đồng thời cần xác minh payload thật của AGY CLI (chạy 1 phiên `agy` thật, capture stdin) để chốt semantics — ghi kết quả vào research doc.

**P1-4. AGY: Reset ETA luôn hiện `XhOm` (phút = 0)**
`fmt_sec_eta`: `printf '%dh%dm' "$h"` — thiếu tham số `$m` (printf shell điền 0 cho specifier thiếu arg). Đã tái hiện: `printf '%dh%dm' 1` → `1h0m`. Test hiện tại pass vì 3600s tình cờ đúng là `1h0m`.
- **Fix:** `printf '%dh%dm' "$h" "$m"`. Sửa test dùng 5400s và assert `1h30m`.
